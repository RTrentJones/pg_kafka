# pg_kafka code audit — June 2026

A comprehensive, line-by-line audit of pg_kafka and the tracking record for its remediation.

- **Method:** 8 parallel deep readers, one per subsystem (protocol codec · runtime/dispatch · API
  handlers · consumer-group coordinator + assignment · Postgres storage · shadow-core · shadow-replay/
  license · config/error/tests). Each read its files in full and cross-checked call sites. ~51k lines
  covered (≈32k `src/` + ≈19k `kafka_test/`). The audit is **static** (no build/run); a few items that
  depend on `kafka-protocol 0.17` internals are marked *needs confirmation*.
- **Reconciliation:** the readers audited tree `d61b637` (17 commits behind `main` at the time). Only 4
  source files differed, and they were exactly this period's two fixes — **LeaveGroup v3+ `members[]`**
  (PR #18) and **Fetch capped to v11** (PR #19) — so those two findings are recorded as already-fixed
  and everything else stands against `main`. Line numbers below are as-audited; small shifts (≤ a few
  lines) are possible in `worker.rs`, `handlers/coordinator.rs`, and `response_builders.rs`.
- **Status legend:** 🔴 Open · 🟡 In progress · ✅ Fixed (PR #) · 🔵 Accepted / by-design (rationale) ·
  ⚪ Deferred.

This document is the **single source of truth** for remediation progress. Each fix PR flips its rows to
✅ and appends to the [resolution changelog](#resolution-changelog).

## Architecture in one screen

**A single-writer core.** A Postgres bgworker binds TCP on the main thread and spawns **one** tokio
network thread for I/O, framing (`LengthDelimitedCodec`, 100 MB cap), and codec. Every decoded request
is forwarded over a **bounded crossbeam channel (cap 10,000)** to **one synchronous DB thread**, which
runs all SPI inside a per-request `transaction()` + nested subtransaction. Responses flow back through
per-request channels, drained in **request-arrival order** so per-connection FIFO holds.

This single-writer decision is the spine of the design: it makes the "shared" store and the
`Arc<RwLock>` coordinator **effectively race-free** (network threads never call handlers), at the cost
of throughput — and it makes *any* blocking call on the DB thread a **global stall**.

- **Codec** delegates per-API decode/encode to the `kafka-protocol` crate; bespoke surface is the
  dispatch match, the request/response enums (`messages.rs`), a legacy MessageSet fallback, and the
  ApiVersions table.
- **Coordinator** is an in-memory group state machine (Empty→PreparingRebalance→CompletingRebalance→
  Stable); only committed offsets persist.
- **Storage** is one relational schema — a single `kafka.messages` table keyed by `(topic_id,
  partition_id, partition_offset)`, sibling tables for offsets/idempotence/txn. Offset assignment =
  `pg_advisory_xact_lock` + `MAX(offset)+1`. **SQL is fully parameterized — no injection found.**
- **Shadow mode** is a `ShadowStore<PostgresStore>` decorator that dual-writes/forwards to a real
  broker for migration. Large parts are incomplete or unwired.

## Findings in detail

### Security / DoS — the untrusted-input boundary

The TCP listener parses attacker-controlled bytes; this is the weakest area.

- **SEC-1** `recordbatch.rs:198–206` — after `message_size` passes `message_size < 0 || remaining <
  message_size`, the code unconditionally reads 6 bytes (CRC + magic + attrs); a 12-byte `records`
  payload survives the guard then panics on `get_u32`. Reachable via any Produce that falls to the
  legacy MessageSet path.
- **SEC-2** `decoding.rs:45` — `decode_request_header_from_buffer` panics rather than returning `Err`
  on a short/empty frame (the original `test_parse_*_header_panics` tests asserted the panic). Fixed by
  wrapping the whole parse in `catch_unwind` so the boundary degrades to a CORRUPT_MESSAGE response; the
  two tests now assert a graceful `Err`. (Note SEC-9: `catch_unwind` only contains *unwinding* panics.)
- **SEC-3** `listener.rs` — the accept loop `tokio::spawn`s unboundedly (no semaphore) and each frame
  may be `MAX_REQUEST_SIZE = 100 MB`. Memory ≈ connections × 100 MB → trivial OOM / fd exhaustion.
- **SEC-4** `listener.rs:499,582` — blocking crossbeam `send()` from connection tasks; with only 2
  tokio workers, a full 10k channel lets two stuck sends starve the runtime. `worker.rs:838,896` do a
  blocking `notify_tx.send()` on the DB thread (should be `try_send`, as the shadow path already does).
- **SEC-5** `recordbatch.rs:139` — `RecordBatchDecoder::decode` auto-decompresses gzip/snappy/lz4/zstd
  with no output cap; the 100 MB limit bounds input only. *(needs confirmation re: crate internals.)*
- **SEC-6** `subscription.rs:87`, `member_assignment.rs:114` — `Vec/HashMap::with_capacity(topics_len)`
  where `topics_len` is validated only `>= 0`; a 6-byte JoinGroup claiming `0x7FFFFFFF` → multi-GB alloc.
- **SEC-7** `constants.rs:261` — `DEFAULT_HOST="0.0.0.0"` + no wire auth. Anyone reaching `:9092` can
  produce/consume and run DeleteTopics/DeleteGroups. (SASL is an explicit non-goal; this is about the
  *default* and documentation.)
- **SEC-8** `config.rs:445` — `SHADOW_SASL_PASSWORD` uses default GUC flags, so any DB role can `SHOW`
  the cleartext. Only the license key got `SUPERUSER_ONLY|NO_SHOW_ALL`.
- **SEC-9** *(found during PR #21 fuzzing — the most severe item in this audit)* — the `kafka-protocol`
  crate's per-API body decoders call `Vec::with_capacity(n)` on an `n` read straight from a wire array-
  length prefix, with no check against remaining bytes. A single crafted request (the fuzzer hit a Fetch
  with a huge `topics_len`) makes the decoder attempt a ~180 GB allocation, which calls
  `handle_alloc_error` → **`abort()` (SIGABRT)**. This is an allocation failure, **not** an unwinding
  panic, so `parse_request`'s `catch_unwind` cannot contain it: one small unauthenticated packet aborts
  the entire worker process. SEC-6 (our own pre-alloc caps) does not help because the allocation happens
  inside the dependency. Fix requires a pre-decode bound on array-length fields and/or a `kafka-protocol`
  upgrade to a version that uses `try_reserve`/bounded decoding; the full-pipeline fuzz harness is
  restored as the regression guard once this lands.

### Correctness / durability

- **BUG-1** `constants.rs` — **fixed in PR #22**, verified against Apache Kafka's `Errors`:
  `OUT_OF_ORDER_SEQUENCE` 47→**45**, `CONCURRENT_TRANSACTIONS` 53→**51**, `INVALID_TXN_STATE` 90→**48**,
  `PRODUCER_FENCED` 91→**90**, `TRANSACTIONAL_ID_NOT_FOUND` 79→**105** (a 5th the audit missed). Also
  removed the bogus `ERROR_TRANSACTION_TIMED_OUT=94` (94 is `INCONSISTENT_VOTER_SET`; Kafka has no
  timed-out code) and mapped `TransactionTimedOut`→`PRODUCER_FENCED`. They ship via
  `error.rs::to_kafka_error_code()`, so a retryable out-of-order sequence was reported as a fatal epoch
  fence.
- **BUG-2** `recordbatch.rs:204` — CRC explicitly skipped; corrupt batches accepted silently.
- **BUG-3** `postgres.rs:149,1660` — `partition_offset = MAX+1`; deleting aborted tail rows makes the
  next produce **reissue the same offsets** and moves the HWM backward. Violates offset monotonicity.
- **BUG-4** `postgres.rs:316` — HWM = `MAX(partition_offset)+1` with no `txn_state` filter (LSO at
  `:1573` is correct).
- **BUG-5** `worker.rs:601` — subtransaction rollback keeps the worker alive but `response_tx` is
  dropped unsent → the client hangs to its own timeout instead of `UNKNOWN_SERVER_ERROR`.
- **BUG-6** `consumer.rs:228` — fetch-all OffsetFetch logs and falls through to `error_code=NONE`.
- **BUG-7** `postgres.rs:176,262` — `Record.timestamp` never stored; `created_at` is `TIMESTAMP` (no
  tz) read via `EXTRACT(EPOCH …)`, TZ-skewed on non-UTC servers.
- **BUG-8** `partitioner.rs:33` — `% partition_count` guarded only by `debug_assert!`.
- **BUG-9** `coordinator.rs:194` — `is_leader = members.is_empty()`; a real leader rejoining a
  rebalance gets `is_leader=false` + empty member list. Masked because SyncGroup recomputes server-side.

### Shadow mode

Shadow mode is a fire-and-forget mirror to a real broker for migration (reads pass through; writes
persist locally and forward). The docs imply a "compare pg_kafka vs production" validator — **that
comparison logic does not exist**, and the live path records nothing.

- **SH-1** `config.rs:220` — `validate()` only called by tests; clamping/normalization dead at runtime.
- **SH-2** `primary.rs:90` — first `pg_is_in_recovery()` cached forever; promoted standby never forwards.
- **SH-3** `store.rs:735` — reload `update()`s without `clear()`; deleted rows stay cached.
- **SH-4** `store.rs:1449` — timeout sweep never removes `pending_txn_messages` entries → RAM leak.
- **SH-5** `store.rs:324` — `is_healthy()` blocking metadata fetch per produce (per-record on commit).
- **SH-6** `store.rs:935` — ExternalOnly returns `Ok(0)` and skips local storage.
- **SH-7** `store.rs:946` — `forwarded` incremented twice in ExternalOnly non-txn.
- **SH-8** `store.rs:565` — `forward_records_required` never samples `forward_percentage`.
- **SH-9** `store.rs:483,490` — async path: successful forwards invisible to metrics; `try_send` drops
  silently on a full 10k channel.
- **SH-10** `replay.rs:283` — query uses `st.partition_offset`/`st.forwarded` but the table has
  `local_offset`/`external_offset`; replay also never advances progress → re-forwards everything.
- **SH-11** `forwarder.rs` — `ShadowForwarder`/`ReplayEngine` re-exported but never instantiated; the
  documented `kafka.replay_to_shadow()` SQL doesn't exist.
- **SH-12** `producer.rs:104` — `enable.idempotence=false` with retries and default in-flight → reorder/
  dupe within a partition.
- **SH-13** `store.rs:935` — ExternalOnly partial batch failure forwards some, then writes **all**
  locally → double-consume.
- **SH-14** `store.rs:496` — `send_sync` `block_on` on the single DB thread; broker down → ~2-min stall
  freezing all clients.
- **SH-15** `store.rs:533` — forwards pin the local `partition_id`; misroutes if the external topic has
  a different partition count.
- **SH-16** `license.rs` — `check_license()` only warns; `is_licensed()` never consulted; `"a:b"` is
  "valid" (no signature/expiry).

## What's genuinely strong

- **The single-writer model** makes the store and coordinator race-free by construction.
- **SQL is fully parameterized** — no injection; the single-table-keyed-by-`topic_id` schema avoids
  per-topic identifier interpolation.
- **Subtransaction rollback** (`run_request_in_subtransaction`) contains handler panics / Postgres
  ERRORs while keeping the one worker alive and discarding partial writes.
- **Error-vs-panic discipline in handlers** is good; the typed-error→code mapping is exhaustive
  (wildcard-free) and leaks no internal text to the wire.
- **Happy-path E2E is high quality** — exact message counts, gap-free offsets, watermarks, committed
  offsets; idempotent dedup, txn atomicity/isolation, group lifecycle exercised end-to-end.
- **Shadow DualWrite isolation is correct** — local write first via `?`; forwarding is best-effort and
  never corrupts the client's real produce.
- **The partitioner** correctly mirrors Kafka's `toPositive(murmur2(key)) & 0x7fffffff % n`.
