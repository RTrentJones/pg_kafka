## Architecture in one screen

**A single-writer core.** A Postgres bgworker binds TCP on the main thread and spawns **one** tokio
network thread for I/O, framing (`LengthDelimitedCodec`, 100 MB cap), and codec. Every decoded request
is forwarded over a **bounded crossbeam channel (cap 10,000)** to **one synchronous DB thread**, which
runs all SPI inside a per-request `transaction()` + nested subtransaction. Responses flow back through
per-request channels, drained in **request-arrival order** so per-connection FIFO holds.

This single-writer decision is the spine of the design: it makes the "shared" store and the
`Arc<RwLock>` coordinator **effectively race-free** (network threads never call handlers), at the cost
of throughput вАФ and it makes *any* blocking call on the DB thread a **global stall**.

- **Codec** delegates per-API decode/encode to the `kafka-protocol` crate; bespoke surface is the
  dispatch match, the request/response enums (`messages.rs`), a legacy MessageSet fallback, and the
  ApiVersions table.
- **Coordinator** is an in-memory group state machine (EmptyвЖТPreparingRebalanceвЖТCompletingRebalanceвЖТ
  Stable); only committed offsets persist.
- **Storage** is one relational schema вАФ a single `kafka.messages` table keyed by `(topic_id,
  partition_id, partition_offset)`, sibling tables for offsets/idempotence/txn. Offset assignment =
  `pg_advisory_xact_lock` + `MAX(offset)+1`. **SQL is fully parameterized вАФ no injection found.**
- **Shadow mode** is a `ShadowStore<PostgresStore>` decorator that dual-writes/forwards to a real
  broker for migration. Large parts are incomplete or unwired.

## Findings in detail

### Security / DoS вАФ the untrusted-input boundary

The TCP listener parses attacker-controlled bytes; this is the weakest area.

- **SEC-1** `recordbatch.rs:198вАУ206` вАФ after `message_size` passes `message_size < 0 || remaining <
  message_size`, the code unconditionally reads 6 bytes (CRC + magic + attrs); a 12-byte `records`
  payload survives the guard then panics on `get_u32`. Reachable via any Produce that falls to the
  legacy MessageSet path.
- **SEC-2** `decoding.rs:45` вАФ `decode_request_header_from_buffer` panics rather than returning `Err`
  on a short/empty frame (the original `test_parse_*_header_panics` tests asserted the panic). Fixed by
  wrapping the whole parse in `catch_unwind` so the boundary degrades to a CORRUPT_MESSAGE response; the
  two tests now assert a graceful `Err`. (Note SEC-9: `catch_unwind` only contains *unwinding* panics.)
- **SEC-3** `listener.rs` вАФ the accept loop `tokio::spawn`s unboundedly (no semaphore) and each frame
  may be `MAX_REQUEST_SIZE = 100 MB`. Memory вЙИ connections √Ч 100 MB вЖТ trivial OOM / fd exhaustion.
- **SEC-4** `listener.rs:499,582` вАФ blocking crossbeam `send()` from connection tasks; with only 2
  tokio workers, a full 10k channel lets two stuck sends starve the runtime. `worker.rs:838,896` do a
  blocking `notify_tx.send()` on the DB thread (should be `try_send`, as the shadow path already does).
- **SEC-5** `recordbatch.rs:139` вАФ `RecordBatchDecoder::decode` auto-decompresses gzip/snappy/lz4/zstd
  with no output cap; the 100 MB limit bounds input only. *(needs confirmation re: crate internals.)*
- **SEC-6** `subscription.rs:87`, `member_assignment.rs:114` вАФ `Vec/HashMap::with_capacity(topics_len)`
  where `topics_len` is validated only `>= 0`; a 6-byte JoinGroup claiming `0x7FFFFFFF` вЖТ multi-GB alloc.
- **SEC-7** `constants.rs:261` вАФ `DEFAULT_HOST="0.0.0.0"` + no wire auth. Anyone reaching `:9092` can
  produce/consume and run DeleteTopics/DeleteGroups. (SASL is an explicit non-goal; this is about the
  *default* and documentation.)
- **SEC-8** `config.rs:445` вАФ `SHADOW_SASL_PASSWORD` uses default GUC flags, so any DB role can `SHOW`
  the cleartext. Only the license key got `SUPERUSER_ONLY|NO_SHOW_ALL`.
- **SEC-9** *(found during PR #21 fuzzing вАФ the most severe item in this audit)* вАФ the `kafka-protocol`
  crate's per-API body decoders call `Vec::with_capacity(n)` on an `n` read straight from a wire array-
  length prefix, with no check against remaining bytes. A single crafted request (the fuzzer hit a Fetch
  with a huge `topics_len`) makes the decoder attempt a ~180 GB allocation, which calls
  `handle_alloc_error` вЖТ **`abort()` (SIGABRT)**. This is an allocation failure, **not** an unwinding
  panic, so `parse_request`'s `catch_unwind` cannot contain it: one small unauthenticated packet aborts
  the entire worker process. SEC-6 (our own pre-alloc caps) does not help because the allocation happens
  inside the dependency. Fix requires a pre-decode bound on array-length fields and/or a `kafka-protocol`
  upgrade to a version that uses `try_reserve`/bounded decoding; the full-pipeline fuzz harness is
  restored as the regression guard once this lands.

### Correctness / durability

- **BUG-1** `constants.rs` вАФ **fixed in PR #22**, verified against Apache Kafka's `Errors`:
  `OUT_OF_ORDER_SEQUENCE` 47вЖТ**45**, `CONCURRENT_TRANSACTIONS` 53вЖТ**51**, `INVALID_TXN_STATE` 90вЖТ**48**,

### Shadow mode

Shadow mode is a fire-and-forget mirror to a real broker for migration (reads pass through; writes
persist locally and forward). The docs imply a "compare pg_kafka vs production" validator вАФ **that
comparison logic does not exist**, and the live path records nothing.

- **SH-1** `config.rs:220` вАФ `validate()` only called by tests; clamping/normalization dead at runtime.
- **SH-2** `primary.rs:90` вАФ first `pg_is_in_recovery()` cached forever; promoted standby never forwards.
- **SH-3** `store.rs:735` вАФ reload `update()`s without `clear()`; deleted rows stay cached.
- **SH-4** `store.rs:1449` вАФ timeout sweep never removes `pending_txn_messages` entries вЖТ RAM leak.
- **SH-5** `store.rs:324` вАФ `is_healthy()` blocking metadata fetch per produce (per-record on commit).
- **SH-6** `store.rs:935` вАФ ExternalOnly returns `Ok(0)` and skips local storage.
- **SH-7** `store.rs:946` вАФ `forwarded` incremented twice in ExternalOnly non-txn.
- **SH-8** `store.rs:565` вАФ `forward_records_required` never samples `forward_percentage`.
- **SH-9** `store.rs:483,490` вАФ async path: successful forwards invisible to metrics; `try_send` drops
  silently on a full 10k channel.
- **SH-10** `replay.rs:283` вАФ query uses `st.partition_offset`/`st.forwarded` but the table has
  `local_offset`/`external_offset`; replay also never advances progress вЖТ re-forwards everything.
- **SH-11** `forwarder.rs` вАФ `ShadowForwarder`/`ReplayEngine` re-exported but never instantiated; the
  documented `kafka.replay_to_shadow()` SQL doesn't exist.
- **SH-12** `producer.rs:104` вАФ `enable.idempotence=false` with retries and default in-flight вЖТ reorder/
  dupe within a partition.
- **SH-13** `store.rs:935` вАФ ExternalOnly partial batch failure forwards some, then writes **all**
  locally вЖТ double-consume.
- **SH-14** `store.rs:496` вАФ `send_sync` `block_on` on the single DB thread; broker down вЖТ ~2-min stall
  freezing all clients.
- **SH-15** `store.rs:533` вАФ forwards pin the local `partition_id`; misroutes if the external topic has
  a different partition count.
- **SH-16** `license.rs` вАФ `check_license()` only warns; `is_licensed()` never consulted; `"a:b"` is
  "valid" (no signature/expiry).

## What's genuinely strong

- **The single-writer model** makes the store and coordinator race-free by construction.
- **SQL is fully parameterized** вАФ no injection; the single-table-keyed-by-`topic_id` schema avoids
